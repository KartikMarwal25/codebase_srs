What makes it non-generic
Unlike a basic chatbot or RAG wrapper, CodeGuard has a genuine multi-agent architecture: a supervisor orchestrator delegates to four specialized agents running in parallel via asyncio.gather(), each returning a typed Pydantic object. An assembler agent deduplicates findings and assigns severity scores. The system stores a persistent review history in PostgreSQL and ingests real GitHub repositories — not toy code snippets.

System architecture
High-level view of all system components and how they connect — from user input through to the final report and persistent storage.
Backend layer
FastAPI handles all incoming requests asynchronously. A GitHub file fetcher pulls .py files from repo URLs. A chunker splits large files using Python AST to preserve function boundaries before sending to agents.
Agent layer
A LangGraph StateGraph manages state across the pipeline. The orchestrator node routes code to all four agents simultaneously. Each agent uses LangChain with Pydantic structured output — no raw text parsing.
Agent pipeline detail
Each agent's internal structure, what it receives, what Pydantic model it returns, and how the assembler merges everything.
Four specialized agents
Bug finder agent
Receives: code chunk + filename
Analyzes: null dereferences, index errors, off-by-one, unhandled exceptions, dead code
Returns: BugFinderOutput(issues: List[Issue]) where Issue has title, severity, line_number, suggestion
Security agent
Receives: code chunk + filename
Analyzes: SQL injection, hardcoded secrets, insecure deserialization, open redirects, path traversal
Returns: SecurityOutput(vulnerabilities: List[Vulnerability]) with CVE references where applicable
Style checker agent
Receives: code chunk
Analyzes: PEP 8 violations, naming conventions, function complexity, magic numbers, missing docstrings
Returns: StyleOutput(issues: List[StyleIssue]) with category and fix suggestion
Test generator agent
Receives: code chunk (function-level)
Analyzes: missing test coverage, edge cases, boundary conditions
Returns: TestOutput(test_cases: List[TestCase]) with actual runnable pytest code per function
Assembler agent logic
The assembler agent receives all four Pydantic outputs. Its job: (1) deduplicate findings that appear in more than one agent's output for the same line, (2) assign a unified severity score (CRITICAL / HIGH / MEDIUM / LOW) to every finding, (3) rank findings by severity, (4) produce the final ReviewReport Pydantic model with metadata — total issues count, risk score (0–100), and per-agent summaries.
LangGraph state definition
class CodeReviewState(TypedDict):
    code: str
    filename: str
    repo_url: Optional[str]
    bug_results: Optional[BugFinderOutput]
    security_results: Optional[SecurityOutput]
    style_results: Optional[StyleOutput]
    test_results: Optional[TestOutput]
    final_report: Optional[ReviewReport]
Why asyncio.gather matters
Each agent call takes ~7–10 seconds. Running sequentially = 28–40 seconds per review. Running with asyncio.gather() = 7–10 seconds total. This single architectural decision is the difference between a demo that impresses and one that times out. Every agent node must be an async function.
Request flow — step by step
What happens from the moment a user submits code to the moment the report appears on screen.
1
User submits via Streamlit
Pastes code or enters GitHub repo URL. Clicks "Review". Streamlit calls POST /review on the FastAPI backend.
2
FastAPI validates input
Checks if input is raw code or GitHub URL. Validates URL format. Returns 422 immediately for invalid input. Creates a review_id (UUID).
3
GitHub file fetcher (if URL provided)
PyGithub pulls all .py files recursively. Handles rate limits with exponential backoff. Skips non-Python files and binary files. Returns list of (filename, code_content) tuples.
4
Code chunker
Parses code with ast.parse(). Splits by function/class definitions (not by character count). Each chunk stays within token budget (≈3000 tokens). Large functions split at logical boundaries.
5
LangGraph orchestrator starts
Initializes CodeReviewState with code + filename. Orchestrator node logs start time, token budget, and chunk count. Routes to all four agent nodes simultaneously.
6
Four agents run via asyncio.gather()
Bug finder, security, style, and test-gen agents execute in parallel. Each calls LLM with structured output prompt. Each returns a typed Pydantic object. Failed agents return an empty result (not an error).
7
Assembler agent merges results
Deduplicates findings by (line_number, issue_type). Assigns unified severity (CRITICAL/HIGH/MEDIUM/LOW). Computes risk score 0–100. Produces final ReviewReport object.
8
PostgreSQL storage
Stores: review_id, repo_url, filename, created_at, total_issues, risk_score, full report JSON. Async SQLAlchemy session — non-blocking. Review is queryable via GET /reviews.
9
Response to Streamlit
FastAPI returns ReviewReport JSON. Streamlit renders it in 4 tabs (Bugs / Security / Style / Tests). Severity color coded: red = CRITICAL, amber = HIGH, gray = LOW. Download button exports Markdown.
Tech stack — Stage 1
Every library and tool you will use, why it's there, and which part of the JD it covers.
Core agent layer (new to learn)
LangChain
LCEL pipe syntax for building chains. ChatPromptTemplate for system/human prompts. with_structured_output() for Pydantic output. Core of every agent.
LangGraph
StateGraph for multi-agent orchestration. Nodes = Python async functions. Edges = control flow. Conditional edges = orchestrator routing logic. The backbone of the pipeline.
Pydantic v2
BaseModel for typed agent outputs. Eliminates raw text parsing. Every agent returns a structured Pydantic object — not a string.
OpenAI GPT-4o-mini
Primary LLM. Cheap (~$0.15/1M tokens) and fast. Sufficient for all four agents. Swappable to Claude/Groq later.
Backend (extend existing knowledge)
FastAPI
You know this already. Need to extend with: async/await, BackgroundTasks, asyncio.gather(), SSE for streaming. POST /review, GET /reviews endpoints.
SQLAlchemy (async)
AsyncSession + asyncpg driver. Non-blocking DB writes. You know SQL — SQLAlchemy is just a Python wrapper.
asyncpg
Fast async PostgreSQL driver. Used under SQLAlchemy. Install: pip install asyncpg.
PyGithub
Fetch all .py files from a GitHub repo URL. repo.get_contents() recursive traversal. Personal access token auth.
Frontend + infra (new, but easy)
Streamlit
4-tab report UI. 1 day to learn. st.tabs(), st.download_button(), st.spinner(), st.sidebar for review history.
PostgreSQL
You know SQL. Run locally via Docker. Deploy as Render PostgreSQL service (free tier).
Docker + docker-compose
Dockerfile for FastAPI. docker-compose.yml launches app + postgres together locally. One command: docker-compose up.
Render
Deployment platform. Free tier. Connect GitHub repo → auto-deploy. Add env vars (OPENAI_API_KEY, DATABASE_URL) in dashboard.

Stage 2 — Technical depth
Only start these after Stage 1 is deployed with a live URL. These upgrades take the project from 7.5/10 to 8.5/10 by adding real engineering underneath the LLM layer.
Do not touch Stage 2 until your live Render URL works and you have a demo GIF in your README. Every hour on Stage 2 before that is wasted.
A — AST-based bug detection
Python AST pre-analysis
Before sending code to the bug finder LLM, parse it with Python's built-in ast module. Extract: all function names + args, return types, exception handlers, unused variables, unreachable code. Pass this structured metadata to the LLM as context. Now the LLM has structural facts, not just raw text. This is the difference between "prompting" and "hybrid analysis".
ast.parse()ast.walk()ast.FunctionDefLibCST (optional)Tree-sitter (multi-lang)
B — Static analysis integration
Bandit + Semgrep as pre-processors
Run Bandit (security linter) and optionally Semgrep via subprocess before the security agent LLM call. Parse their JSON output. Feed structured findings into the security agent as context. The LLM then reasons about real, confirmed findings — not guessing. This is hybrid static + LLM analysis. This one change alone is what makes it "real engineering" vs "prompt engineering".
bandit -r . -f jsonsemgrep --jsonsubprocess.run()pylint (style)mypy (type check)
C — Semantic code chunking
Split by function boundary, not character count
Upgrade the chunker to use ast.parse() to find all function and class definitions. Each chunk = one function or class. Preserves context that character-count chunking destroys. Also adds a class-level context header to each chunk so the LLM knows what module it's looking at.
D — Repo memory with Mem0
Track patterns across reviews
Add Mem0 (open-source memory library) to track repeated issues per repository. After each review, store findings in Mem0 keyed by repo_url. Before the next review of the same repo, retrieve past findings and pass as context. Agents now know "this repo has had SQL injection issues in auth module 3 times" — that is real contextual intelligence.

Stage 3 — Serious engineering
Long-term upgrades. Do not start until Stage 2 is complete. These take the project to 9/10 territory and make it industry-level.
Autonomous fix loop
Detect bug → generate patch → apply to temp file → run pytest → if pass: commit suggestion. If fail: retry with error context (max 2 retries). This is what "agentic" really means — not just detection but action.
GitHub PR auto-review
GitHub webhook triggers on pull_request event. Your backend receives the diff. Agents review only the changed files. Post findings as PR review comments via GitHub API. This is what CodeRabbit and GitHub Copilot do.
Execution sandbox
Run generated test cases in a Docker container with a timeout. Parse pytest output. Verify that generated tests actually pass. Gives a "test validity" signal alongside test suggestions.
Dependency graph reasoning
Build import graph of the repo. Identify which modules depend on each other. Pass dependency context to agents. A bug in a utility function now triggers review of all files that import it.
Stage 3 is ambitious. Even implementing one of these (the autonomous fix loop is most impressive) takes CodeGuard to a level very few student projects reach. Prioritize Stage 1 and Stage 2 first.
